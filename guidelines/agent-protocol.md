# Agent Protocol

Standard conventions for nrworkflow agents.

## Ticket Header

Every agent prompt MUST include a ticket header at the top:

```markdown
# {Agent Name} - {ticket_id}

## Agent: {agent_type}
## Ticket: {ticket_id}
## Parent Session: {parent_session_id}
## Child Session: {child_session_id}
## Model ID: {model_id}
```

The spawner injects these values automatically using placeholders:
- `${AGENT}` - Agent type (e.g., "setup-analyzer", "implementor")
- `${TICKET_ID}` - Current ticket ID
- `${PARENT_SESSION}` - Parent session UUID (from skill)
- `${CHILD_SESSION}` - Child session UUID (generated by spawner)
- `${PROJECT_SPECIFIC}` - Project override content (or empty)
- `${MODEL_ID}` - Full model identifier in cli:model format (e.g., "claude:sonnet")
- `${MODEL}` - Just the model name (e.g., "sonnet")

## Session Tracking

### Parent Session
The main Claude session that runs the skill:
1. User starts Claude via wrapper (e.g., `cld()`) which generates UUID
2. UUID is injected as `SESSION_MARKER=<uuid>` in context
3. Skill passes marker to spawner: `nrworkflow agent spawn <type> <ticket> --session=<uuid> -w <name>`
4. Spawner injects into agent prompt as `${PARENT_SESSION}`

Both `--session` and `-w/--workflow` are **required** for `agent spawn`:
- `--session`: Links spawned agents to the orchestrating session
- `-w/--workflow`: Specifies which workflow to use (required since tickets can have multiple workflows)

**Note:** Project is auto-discovered from `.claude/nrworkflow/config.json` (searched upward) or `NRWORKFLOW_PROJECT` env var.

### Child Session
The spawned agent session:
1. Spawner generates UUID: `child_session_id = uuid.uuid4()`
2. Passed to CLI: `--session-id <child_uuid>`
3. Injected into prompt as `## Child Session: <child_uuid>`
4. Stored in ticket state for tracking/kill support (per workflow)

## Completion Commands

Every agent MUST call a completion command when done. The `--workflow` flag is **required**:

### Successful Completion
```bash
nrworkflow agent complete ${TICKET_ID} ${AGENT} --workflow=${WORKFLOW}

# For parallel agents, include the model ID:
nrworkflow agent complete ${TICKET_ID} ${AGENT} --workflow=${WORKFLOW} --model=${MODEL_ID}
```

### Failed Completion
```bash
nrworkflow agent fail ${TICKET_ID} ${AGENT} --workflow=${WORKFLOW} --reason="<explanation of what went wrong>"

# For parallel agents:
nrworkflow agent fail ${TICKET_ID} ${AGENT} --workflow=${WORKFLOW} --model=${MODEL_ID} --reason="..."
```

The agent type is validated against the active agent for safety.
The workflow is passed by the spawner and should be available in the agent's context.

The spawner reads the result from ticket state (for the specific workflow) after the agent process exits:
- `result=pass` -> phase result = pass
- `result=fail` -> phase result = fail
- No result set -> falls back to checking output for `AGENT_STATUS:` markers

### Parallel Agent Completion

For parallel agents (multiple agents running the same phase with different models):
1. Each agent calls completion with its `--model` flag
2. The phase completes when ALL parallel agents have finished
3. The phase passes only if ALL agents passed
4. If any agent fails, the phase fails

## Context Limits

Agents should be mindful of context limits:
- Read only necessary files
- Use findings from previous phases (don't re-explore)
- Store findings incrementally (don't wait until end)

## Tool Usage

Agents typically have access to:
- `Bash` - Run commands (build, test, etc.)
- `Read` - Read files
- `Edit` - Edit existing files
- `Write` - Create new files
- `Glob` - Find files by pattern
- `Grep` - Search file contents

Tool list is configurable per project in config.json.

## State Management

Agents interact with nrworkflow state via CLI. All commands require `--workflow/-w`:

```bash
# Store findings (by agent type)
nrworkflow findings add <ticket> <agent-type> <key> <value> -w <workflow>

# For parallel agents, store findings under model key:
nrworkflow findings add <ticket> <agent-type> <key> <value> -w <workflow> --model=${MODEL_ID}

# Read findings from another agent
nrworkflow findings get <ticket> <agent-type> -w <workflow>
nrworkflow findings get <ticket> <agent-type> -w <workflow> --model=<model_id>  # specific model

# Set ticket category (setup-analyzer only)
nrworkflow set <ticket> category <docs|simple|full> -w <workflow>

# Mark completion
nrworkflow agent complete <ticket> <agent-type> -w <workflow>
nrworkflow agent complete <ticket> <agent-type> -w <workflow> --model=${MODEL_ID}  # parallel
nrworkflow agent fail <ticket> <agent-type> -w <workflow> --reason="..."
nrworkflow agent fail <ticket> <agent-type> -w <workflow> --model=${MODEL_ID} --reason="..."  # parallel
```

The workflow name and model ID should be passed to the agent via the spawner context.

## Error Handling

When an agent encounters errors:
1. Log the error clearly
2. Store any partial findings
3. Call `nrworkflow agent fail <ticket> --reason="..."`
4. The spawner will mark the phase as failed

### Server Connection Errors

**CRITICAL**: If you get "nrworkflow server is not running" errors:
- **NEVER start the server yourself** (`nrworkflow serve` is forbidden)
- The server is managed by the orchestrator and must already be running
- If commands fail, call `nrworkflow agent fail` with the error and exit
- The orchestrator will handle server issues

The orchestrating skill can then:
- Retry the agent (if retries available)
- Fail the ticket
- Ask user for guidance

## Completion Checklist

Before calling completion command, verify:
1. All required findings stored
2. Build/tests run (if applicable)
3. Changes are minimal and focused
4. Patterns from investigation followed
5. Acceptance criteria addressed

## Example Agent Flow

```
1. Read ticket header -> extract ticket ID and workflow
2. Read previous agent findings (using -w flag)
3. Do the work (explore, implement, verify, etc.)
4. Store findings incrementally (using -w flag)
5. Verify completion criteria
6. Call: nrworkflow agent complete ${TICKET_ID} ${AGENT} -w ${WORKFLOW}
```
