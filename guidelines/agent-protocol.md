# Agent Protocol

Standard conventions for nrworkflow agents.

## Base Header (Auto-Prepended)

The spawner automatically prepends a base header to all agent templates:

```markdown
## Agent: {agent_type}
## Ticket: {ticket_id}
## Workflow: {workflow_name}
## Model: {model}
## Model ID: {model_id}
## Parent Session: {parent_session_id}
## Child Session: {child_session_id}

---
```

This header provides session and model context for tracking and debugging. You don't need to include it in your templates - the spawner adds it at runtime.

### Template Variables

Templates use placeholders that the spawner injects:
- `${AGENT}` - Agent type (e.g., "setup-analyzer", "implementor")
- `${TICKET_ID}` - Current ticket ID
- `${PARENT_SESSION}` - Parent session UUID (from skill)
- `${CHILD_SESSION}` - Child session UUID (generated by spawner)
- `${WORKFLOW}` - Workflow name (e.g., "feature", "bugfix")
- `${MODEL_ID}` - Full model identifier in cli:model format (e.g., "claude:sonnet")
- `${MODEL}` - Just the model name (e.g., "sonnet")

## Session Tracking

### Parent Session
The main Claude session that runs the skill:
1. User starts Claude via wrapper (e.g., `cld()`) which generates UUID
2. UUID is injected as `SESSION_MARKER=<uuid>` in context
3. Orchestrator spawns agents in-process
4. Spawner injects into agent prompt as `${PARENT_SESSION}`

**Note:** Project is auto-discovered from `.claude/nrworkflow/config.json` (searched upward) or `NRWORKFLOW_PROJECT` env var.

### Child Session
The spawned agent session:
1. Spawner generates UUID: `child_session_id = uuid.uuid4()`
2. Passed to CLI: `--session-id <child_uuid>`
3. Injected into prompt as `## Child Session: <child_uuid>`
4. Stored in ticket state for tracking/kill support (per workflow)

## Completion Commands

**CRITICAL:** Every agent MUST call a completion command when done. The spawner enforces **hybrid completion semantics**:

- Exit code 0 alone is **NOT sufficient** for passing
- Within 60 seconds of exit, the spawner checks for explicit `nrworkflow agent complete`
- If no explicit completion is found, the phase **fails** with reason `no_complete`
- Non-zero exit codes fail immediately with reason `exit_code`
- Timeout results in `SIGTERM` → 5s grace → `SIGKILL` with reason `timeout`

The `--workflow` flag is **required**:

### Successful Completion
```bash
nrworkflow agent complete ${TICKET_ID} ${AGENT} --workflow=${WORKFLOW}

# For parallel agents, include the model ID:
nrworkflow agent complete ${TICKET_ID} ${AGENT} --workflow=${WORKFLOW} --model=${MODEL_ID}
```

### Failed Completion
```bash
nrworkflow agent fail ${TICKET_ID} ${AGENT} --workflow=${WORKFLOW} --reason="<explanation of what went wrong>"

# For parallel agents:
nrworkflow agent fail ${TICKET_ID} ${AGENT} --workflow=${WORKFLOW} --model=${MODEL_ID} --reason="..."
```

The agent type is validated against the active agent for safety.
The workflow is passed by the spawner and should be available in the agent's context.

The spawner reads the result from ticket state (for the specific workflow) after the agent process exits:
- `result=pass` -> phase result = pass
- `result=fail` -> phase result = fail
- No result set -> falls back to checking output for `AGENT_STATUS:` markers

### Parallel Agent Completion

For parallel agents (multiple agents running the same phase with different models):
1. Each agent calls completion with its `--model` flag
2. The phase completes when ALL parallel agents have finished
3. The phase passes only if ALL agents passed
4. If any agent fails, the phase fails

## Context Continuation

When an agent runs out of context window space, it can request continuation instead of failing. The spawner will relaunch the agent with fresh context.

### How It Works
1. Agent detects it is running low on context (or is signaled by the hook script)
2. Agent stores all intermediate progress as findings, including a `continuation_notes` key
3. Agent calls `nrworkflow agent continue <ticket> <agent-type> -w <workflow>`
4. Spawner receives the `"continue"` result and relaunches the agent
5. New agent reads `continuation_notes` from findings to pick up where the previous one left off
6. The session is linked via `ancestor_session_id` for chain tracking

### Limits
- `max_continuations` (default: 3) — max times an agent can be continued per spawn
- If the limit is reached, the agent is marked as failed with reason `max_continuations`
- Each continuation creates a new session linked to the ancestor

### Configuration
In `config.json`:
```json
{
  "spawner": {
    "max_continuations": 3,
    "context_threshold": 85
  }
}
```

### Continuation Command
```bash
# Signal that context is exhausted and task needs fresh context
nrworkflow agent continue ${TICKET_ID} ${AGENT} -w ${WORKFLOW}

# For parallel agents:
nrworkflow agent continue ${TICKET_ID} ${AGENT} -w ${WORKFLOW} --model=${MODEL_ID}
```

### Before Continuing
Always store your progress first:
```bash
nrworkflow findings add <ticket> <agent-type> continuation_notes '<what to do next>' -w <workflow>
```

### Context Hook
The `scripts/context-check.sh` hook tracks context usage per session via `/tmp/usable_context.json`. Install it as a Claude Code `PreToolUse` hook for spawned agents. The spawner sets `NRWF_SPAWNED=1` automatically.

## Context Limits

Agents should be mindful of context limits:
- Read only necessary files
- Use findings from previous phases (don't re-explore)
- Store findings incrementally (don't wait until end)
- If context is getting low, request continuation rather than failing

## Tool Usage

Agents typically have access to:
- `Bash` - Run commands (build, test, etc.)
- `Read` - Read files
- `Edit` - Edit existing files
- `Write` - Create new files
- `Glob` - Find files by pattern
- `Grep` - Search file contents

Tool list is configurable per project in config.json.

## State Management

Agents interact with nrworkflow state via CLI. All commands require `--workflow/-w`:

```bash
# Store findings (two syntax modes)
nrworkflow findings add <ticket> <agent-type> <key> <value> -w <workflow>                # single
nrworkflow findings add <ticket> <agent-type> key:'value' [key2:'value2'] -w <workflow>  # multiple

# Append to findings (creates array if needed, flattens arrays)
nrworkflow findings append <ticket> <agent-type> <key> <value> -w <workflow>
nrworkflow findings append <ticket> <agent-type> key:'value' [key2:'value2'] -w <workflow>

# Delete finding keys
nrworkflow findings delete <ticket> <agent-type> <key>... -w <workflow>

# For parallel agents, store findings under model key:
nrworkflow findings add <ticket> <agent-type> <key> <value> -w <workflow> --model=${MODEL_ID}
nrworkflow findings append <ticket> <agent-type> <key> <value> -w <workflow> --model=${MODEL_ID}
nrworkflow findings delete <ticket> <agent-type> <key> -w <workflow> --model=${MODEL_ID}

# Read findings from another agent
nrworkflow findings get <ticket> <agent-type> -w <workflow>                     # all findings
nrworkflow findings get <ticket> <agent-type> -w <workflow> -k <key>            # specific key
nrworkflow findings get <ticket> <agent-type> -w <workflow> --model=<model_id>  # specific model

# Mark completion
nrworkflow agent complete <ticket> <agent-type> -w <workflow>
nrworkflow agent complete <ticket> <agent-type> -w <workflow> --model=${MODEL_ID}  # parallel
nrworkflow agent fail <ticket> <agent-type> -w <workflow> --reason="..."
nrworkflow agent fail <ticket> <agent-type> -w <workflow> --model=${MODEL_ID} --reason="..."  # parallel
nrworkflow agent continue <ticket> <agent-type> -w <workflow>                                 # context continuation
nrworkflow agent continue <ticket> <agent-type> -w <workflow> --model=${MODEL_ID}             # parallel
```

**Append logic:**
- If key doesn't exist: stores value as-is
- If existing is array AND new is array: flattens (merges arrays)
- If existing is array AND new is not array: appends element
- If existing is not array: converts to `[existing, new]`

The workflow name and model ID should be passed to the agent via the spawner context.

### Workflow-Level Findings

Use `workflow` as the agent_type to store data shared across all agents:

```bash
# Store workflow-wide data
nrworkflow findings add <ticket> workflow <key> <value> -w <workflow>

# Read workflow-wide data
nrworkflow findings get <ticket> workflow -w <workflow> -k <key>
```

**When to use workflow vs agent findings:**
- **Agent findings** (`setup-analyzer`, `implementor`, etc.): Phase-specific results that the agent produces
- **Workflow findings** (`workflow`): Cross-cutting data needed by multiple agents (e.g., selected architecture, global constraints)

## Error Handling

When an agent encounters errors:
1. Log the error clearly
2. Store any partial findings
3. Call `nrworkflow agent fail <ticket> --reason="..."`
4. The spawner will mark the phase as failed

### Server Connection Errors

**CRITICAL**: If you get "nrworkflow server is not running" errors:
- **NEVER start the server yourself** (`nrworkflow serve` is forbidden)
- The server is managed by the orchestrator and must already be running
- If commands fail, call `nrworkflow agent fail` with the error and exit
- The orchestrator will handle server issues

The orchestrating skill can then:
- Retry the agent (if retries available)
- Fail the ticket
- Ask user for guidance

## Completion Checklist

Before calling completion command, verify:
1. All required findings stored
2. Build/tests run (if applicable)
3. Changes are minimal and focused
4. Patterns from investigation followed
5. Acceptance criteria addressed

## Example Agent Flow

```
1. Read ticket header -> extract ticket ID and workflow
2. Read previous agent findings (using -w flag)
3. Do the work (explore, implement, verify, etc.)
4. Store findings incrementally (using -w flag)
5. Verify completion criteria
6. Call: nrworkflow agent complete ${TICKET_ID} ${AGENT} -w ${WORKFLOW}
```
